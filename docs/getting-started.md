# Getting started

[← Back to index](../README.md)

## Prerequisites

- A Rust toolchain (stable).
- A Linux Wayland session (wlroots/Sway, KWin/Plasma, or GNOME), **or** access
  to `/dev/input` + `/dev/uinput` for evdev mode.
- For IBus mode: a running `ibus-daemon`.
- For evdev mode: the custom system xkb layout installed (generated by
  `daklak gen-keymap`).

## Build

The daemon crate is `viet-ime-daemon`. Cargo features are only used for
optional integrations (`ibus`, `kde`), declared in
[`crates/daemon/Cargo.toml`](../crates/daemon/Cargo.toml).

```sh
# default build
cargo build -p viet-ime-daemon

# with IBus support
cargo build -p viet-ime-daemon --features ibus

# with KDE Plasma focus/protocol support
cargo build -p viet-ime-daemon --features kde
```

For development installs that need desktop integration, use Meson's debug
buildtype. In this mode the generated KDE desktop entry and IBus component point
to the wrapper scripts in this checkout (`daklak-wrap.sh` and
`daklak-ibus-wrap.sh`). Non-debug Meson builds point to the installed `daklak`
binary.

```sh
# KDE development install
meson setup build --buildtype=debug -Dkde=true
meson install -C build

# GNOME / IBus development install
meson setup build --buildtype=debug -Dibus=true
meson install -C build
```

## Run

The daemon picks its transport at startup. The selection order is:

1. **IBus** — if built with `--features ibus` and IBus mode is enabled
   (via the `--ibus` flag or by being spawned by `ibus-daemon`).
2. **Wayland** — if Wayland is enabled in config.
3. **evdev** — the fallback when Wayland is disabled and evdev grab is enabled.

### wlroots / Sway

Run daklak directly in Wayland mode:

```sh
cargo run -p viet-ime-daemon
```

Development run with logging overrides:

```sh
DAKLAK_LOG_LEVEL=error \
DAKLAK_LOG_MODULES=daklak=debug,viet_ime_wayland_adapter=info \
cargo run -p viet-ime-daemon
```

### KDE Plasma

Build/install the KDE desktop entry, then enable Daklak from Plasma's virtual
keyboard/input-method settings. KWin launches the desktop entry when the input
method is selected.

```sh
meson setup build --buildtype=debug -Dkde=true
meson install -C build
```

### GNOME / IBus

Build/install the IBus component, refresh IBus, then add Daklak as an input
source in GNOME Settings and select it from the input-source switcher. GNOME
launches daklak via IBus when that source is active.

```sh
meson setup build --buildtype=debug -Dibus=true
meson install -C build
ibus write-cache
ibus restart
```

If GNOME Settings does not list Daklak from the user-local component, follow the
copy-to-`/usr/share/ibus/component` notice printed by `meson install`.

### evdev grab

evdev mode is opt-in because it grabs physical keyboards. Install the daklak xkb
layout first (see [Transports](transports.md#evdev)), then enable evdev grab and
disable Wayland for the process:

```sh
DAKLAK_ENABLE_EVDEV_GRAB=1 DAKLAK_ENABLE_WAYLAND=0 \
cargo run -p viet-ime-daemon
```

The same switches can be set in `config.toml` with `enable_evdev_grab = true`
and `enable_wayland = false`.

See [Transports](transports.md) for what each mode requires from the desktop.

## Command-line interface

A few subcommands are handled before the daemon loop starts:

| Command | Effect |
| ------- | ------ |
| `daklak` (no args) | Start the daemon in the auto-selected transport mode. |
| `daklak --ibus` | Force IBus mode (used by the IBus component exec line). |
| `daklak -c ~/.config/daklak/custom.toml` | Use an alternate config file. |
| `DAKLAK_CONFIG=~/.config/daklak/custom.toml daklak` | Same as `--config` via env. |
| `daklak --log-level debug --log-module daklak=debug` | Override logging from the command line. |
| `daklak toggle` | Toggle the running daemon on/off (IPC client). |
| `daklak enable` | Enable composition in the running daemon. |
| `daklak disable` | Disable composition (pass-through). |
| `daklak status` | Print the running daemon's enabled state. |
| `daklak gen-keymap` | Print the daklak synthetic xkb keymap to stdout. |
| `daklak backend` | Print the running daemon's active backend (wayland/ibus/evdev). |
| `daklak backend native` | Switch the running daemon to the native desktop backend. |
| `daklak backend evdev` | Switch the running daemon to the evdev grab backend. |
| `daklak help` | Show usage. |

The control commands (`toggle`/`enable`/`disable`/`status`/`backend`) talk to an
already-running daemon over a Unix socket and work regardless of which transport
mode that daemon is in.

## Tray indicator

When running, daklak exposes a StatusNotifierItem (SNI) tray icon that reflects
and toggles the enabled state. **Left-clicking the icon toggles daklak on/off.**
While off, the icon is the neutral symbolic
keyboard glyph. While on, it shows the Daklak "Đk" badge coloured by the active
backend: **green** for the native (Wayland/IBus) backend and **red** for the
evdev grab backend. When built with `evdev_grab`, the tray also shows an
"Enable evdev" / "Disable evdev" checkbox to switch between native and evdev
backends at runtime.

## Configuration

Configuration is loaded at startup (see [`crates/daemon/src/config.rs`](../crates/daemon/src/config.rs)).
Notable config-file options:

- `DAKLAK_CONFIG=/path/to/file` — alternate config file path (same as `--config`).

Notable logging options:

```toml
log_level = "error"
log_path = "/dev/stdout"
log_modules = ["daklak=debug", "viet_ime_wayland_adapter=info"]
```

- **log level** — base level for all targets (default `error`).
- **log path** — destination for tracing output (default `/dev/stdout`).
- **log modules** — per-target overrides such as `daklak=debug`.
